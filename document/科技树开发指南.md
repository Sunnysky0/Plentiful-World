# Plentiful World 科技树开发指南（HOI4）

## 1. 目的与适用范围

本指南用于规范 Plentiful World 后续科技树开发，目标是：

- 支持“完全自定义键值（不复用原版技术 ID）”
- 保证科技树在科研界面稳定显示
- 保证图标、名称、描述、开局科技、路径关系一致
- 降低“科技不显示/显示错误名称/图标错乱”的返工率

适用目录：

- `common/technologies/*.txt`
- `common/technology_tags/*.txt`
- `common/defines/*.lua`
- `interface/countrytechtreeview.gui`
- `interface/countrytechtreeview.gfx`
- `interface/*.gfx`
- `localisation/*/*.yml`
- `history/countries/*.txt`

---

## 2. HOI4 科技树核心机制（必须理解）

### 2.1 科技定义层（数据层）

技术本体在 `common/technologies/*.txt` 的 `technologies = {}` 内定义。  
每个技术块名就是该技术的 **Tech ID**。

科技分类与科技页签归属在 `common/technology_tags/*.txt` 定义，包含：

- `technology_categories`
- `technology_folders`

### 2.2 界面层（渲染层）

科研树显示由 `interface/countrytechtreeview.gui` 驱动。  
仅在技术文件中写 `folder = {}` 与 `position` 不足以保证显示，GUI 侧必须存在匹配的分支 grid。

### 2.3 图标层（资源层）

技术图标通过 `interface/*.gfx` 中的 `spriteType` 映射。  
命名不匹配会回退默认图或产生显示异常。

### 2.4 本地化层（文本层）

技术名与描述分别读取：

- `<tech_id>`
- `<tech_id>_desc`

缺任一键会显示 key 原文或旧文本。

## 3. 与 The Fire Rises 对齐的实现规则

以下规则来自 3350890356（The Fire Rises）的实际实现，可直接作为本模组标准：

## 3.1 分支 grid 命名规则

**规则：** `gridboxtype.name = "<根科技ID>_tree"`

示例（335）：

- `infantry_weapons1` ↔ `infantry_weapons1_tree`
- `support_weapons` ↔ `support_weapons_tree`
- `attack_helicopter_1` ↔ `attack_helicopter_1_tree`

> 这是科研树是否显示的第一硬条件。  
> 若不匹配，典型日志为：`Found no grid box for tech <tech_id>`。

## 3.2 图标命名规则

**通用图标：** `GFX_<tech_id>_medium`  
**国家特化：** `GFX_<TAG>_<tech_id>_medium`

建议先保证通用键可用，再做国家覆盖。

## 3.3 folder 绑定规则

技术中的：

```txt
folder = { name = infantry_folder ... }
```

必须与 GUI 里的科技页容器一致（如 `infantry_folder` 对应步兵页容器）。

## 3.4 本地化规则

每个技术至少同时提供：

- `<tech_id>`
- `<tech_id>_desc`

并统一语法风格，避免同文件混杂不一致格式。

```txt

@1939 = 4
@1940 = 6
@1942 = 10
```

## 5. 标准开发流程（新增或重构科技树）

按以下顺序执行，不能跳步：

1. **定义分类与页面归属**
   - 在 `common/technology_tags/*.txt` 增加 `technology_categories`
   - 在 `technology_folders` 定义页面（`ledger`、`available`、`doctrine`）
2. **定义技术 ID**
   - 在 `common/technologies/*.txt` 新增技术块
   - `folder.name` 必须指向已存在的 technology folder
   - 规划 `path` / `dependencies`
3. **定义根节点分支 grid**
   - 在 `interface/countrytechtreeview.gui` 增加或修改 `gridboxtype`
   - 命名必须 `<root_tech_id>_tree`
4. **补图标映射**
   - 在 `interface/*.gfx` 新增 `GFX_<tech_id>_medium`
5. **补本地化**
   - 在 `localisation` 增加 `<tech_id>` 与 `<tech_id>_desc`
6. **同步开局科技**
   - `history/countries/*.txt` 的 `set_technology` 改为新 ID
7. **日志验证**
   - 启动后检查 `logs/error.log` 中是否出现关键错误

## 6. 常见错误与快速定位

## 6.1 科技树整页不显示或分支消失

**现象：** 界面空白或整条链不出现。  
**高概率原因：** 根技术 ID 与 `*_tree` 不匹配。  
**日志特征：**

- `Found no grid box for tech <tech_id>`

**处理：**

- 检查根技术 ID（无 `path` 的起点技术）
- 检查 GUI grid 名是否严格为 `<root_tech_id>_tree`

## 6.2 开局科技报错 Unknown effect-type

**现象：** 国家历史文件加载时报技术键异常。  
**原因：** `history/countries` 仍引用旧技术 ID。  
**处理：**

- 全量替换 `set_technology` 为新 ID
- 确认对应技术 ID 已在 `common/technologies` 中存在

## 6.3 图标不对或回退默认

**原因：**

- 缺少 `GFX_<tech_id>_medium`
- 同名 GFX 被其他文件后加载覆盖

**处理：**

- 保证每个技术都有唯一 GFX 键
- 用单一高优先级 `.gfx` 文件维护科技图标，避免重复定义

## 6.4 名称显示旧文本或 key

**原因：**

- 本地化键未同步（尤其是重命名技术 ID 后）
- 多个本地化文件对同一 key 冲突

**处理：**

- 搜索 `<tech_id>` 与 `<tech_id>_desc` 全仓唯一性
- 重启游戏验证缓存

## 6.6 科技名被装备名覆盖（高频误区）

**现象：** 科技界面显示“中正式步枪”等国家装备名，而不是科技本地化名。  
**原因：** 科技使用 `enable_equipments = { infantry_equipment_x }` 时，UI会优先取国家装备命名原型。  
**处理：**

- 步兵科技优先解锁独立装备键（如 `pw_infantry_equipment_1..4`），避免复用原版 `infantry_equipment_*`。
- 同步补齐独立装备的 `GFX_pw_infantry_equipment_*_medium` 与本地化键。

## 6.5 首节点位置偏移

**原因：**

- 年份宏为负值（如 `@1918 = -2`）
- grid 原点与技术坐标体系不一致

**处理：**

- 先将首年锚点校正为 `0`
- 再微调 grid 的 `position`

## 7. 坐标与对齐设计建议

## 7.1 推荐坐标策略

- 主链：`x = 0` 固定，`y` 随年份增长
- 支链：`x` 使用固定偏移（如 `x = 3`）
- 保持同一文件内定义并使用年份宏（`@1936` 等）

## 7.2 先宏后像素

优先调 `@year` 宏，最后才调 GUI `gridboxtype.position`。  
这样能保证全链条相对关系稳定，减少后续调整成本。

## 12. 科技界面 GUI 原理（重点）

## 12.1 页面结构是“容器 + 分支 grid + 项目模板”

`countrytechtreeview.gui` 的基础结构可抽象为：

1. 根窗口：`countrytechtreeview`
2. 页面容器：如 `infantry_folder`、`support_folder`
3. 页面画布容器：常见名 `techtree_stripes`
4. 分支网格：多个 `gridboxtype`（每个对应一个根科技）
5. 项目模板：`techtree_<folder>_item` / `techtree_<folder>_small_item`

结论：科技是否显示，不仅依赖技术定义，还依赖“分支网格 + 项目模板”。

## 12.2 根科技与 grid 的绑定逻辑

- 没有父 `path` 指向它的技术会被视为分支根
- 分支根必须有同名 grid：`<root_tech_id>_tree`
- 子技术通过 `path = { leads_to_tech = ... }` 继承父 grid 并继续排布

这也是 `Found no grid box for tech <tech_id>` 的根本机制来源。

## 12.3 坐标系统如何生效

最终屏幕位置由三层叠加：

1. `gridboxtype.position`：分支局部坐标原点
2. 技术 `folder.position`：格点坐标（`x/y` 单位格）
3. `gridboxtype.slotsize`：单位格像素尺寸（如 `70x70`）

简化公式：

- `最终X ≈ grid_x + tech_x * slot_w`
- `最终Y ≈ grid_y + tech_y * slot_h`

因此：

- 调 `@year` / `folder.position` 是“逻辑坐标调优”
- 调 `gridboxtype.position` / `slotsize` 是“像素级布局调优”

## 12.4 页面标签页（Tab）工作机制

页签由 `folder_tabs` 容器中的 `buttonType` 定义，例如：

- `name = "<folder_name>_tab"`
- `quadTextureSprite = "GFX_<folder_name>_tab"`

点击后切换到同名 folder 页面。  
如果新增页面但没有对应 tab，页面通常无法被正常切换访问。

## 13. 如何新建“科技类型页面”（完整流程）

以下为可执行流程，适用于新建一个完全独立科技页（例如 `pw_biotech_folder`）。

## 13.1 定义 technology folder 与 category

在 `common/technology_tags/*.txt`：

1. 增加 `technology_categories` 条目（如 `pw_biotech_category`）
2. 在 `technology_folders` 增加：

```txt
pw_biotech_folder = {
    ledger = civilian
}
```

若需 DLC 条件，可加 `available = { has_dlc = "xxx" }`。

## 13.2 定义技术并挂载到新页面

在 `common/technologies/*.txt`：

- 技术 `folder.name = pw_biotech_folder`
- 根技术不依赖其他技术
- 子技术通过 `path` 串联

## 13.3 在 GUI 建页面容器

在 `interface/countrytechtreeview.gui`：

1. 新增 `containerWindowType = { name = "pw_biotech_folder" ... }`
2. 在容器内部增加背景、年份文本、分支 `gridboxtype`
3. 为每条根分支增加 `<root_tech_id>_tree`

## 13.4 增加页签按钮

在 `folder_tabs` 容器新增：

```txt
buttonType = {
    name = "pw_biotech_folder_tab"
    position = { x = ... y = 0 }
    quadTextureSprite = "GFX_pw_biotech_folder_tab"
    frame = 1
}
```

## 13.5 增加 tab 贴图与项目样式

在 `interface/countrytechtreeview.gfx`：

- 定义 `GFX_pw_biotech_folder_tab`
- 必要时定义页面背景 `GFX_pw_biotech_techtree_bg`
- 若要独立框体风格，新增：
  - `techtree_pw_biotech_folder_item`
  - `techtree_pw_biotech_folder_small_item`

## 13.6 增加技术图标与本地化

- `interface/*.gfx`：`GFX_<tech_id>_medium`
- `localisation`：`<tech_id>` 与 `<tech_id>_desc`

## 14. 如何“重置原版科技类型”

## 14.1 推荐策略：复制基准 GUI 后再改

对 `countrytechtreeview.gui` 的改造，推荐做法：

1. 复制基准文件到模组 `interface/`
2. 仅在目标 folder 区域做重构
3. 保持其他 folder 原样，降低联动风险

原因：科技界面高度互联，局部不完整改动容易引发隐藏回归。

## 14.2 两种重置方式

### A. 保留原 folder 名，替换其分支内容

适用于不新增页签，只替换该页的科技链。  
示例：保留 `infantry_folder`，但全部根分支改为 `pw_*_tree`。

### B. 新建 folder 名，新增独立页面

适用于新增科技类型。  
必须同时完成：

- `technology_folders` 新增 folder
- GUI 新增同名 container
- `folder_tabs` 新增按钮
- `countrytechtreeview.gfx` 新增 tab sprite

---

## 15. 如何改“科技显示框”的大小与样式

## 15.1 改框体尺寸

在 `countrytechtreeview.gui` 修改对应模板：

- 普通框：`techtree_<folder>_item` 的 `size`
- 小框：`techtree_<folder>_small_item` 的 `size`

并同步调整：

- `Icon` 位置
- `Name` 文本位置/宽度
- `bonus` 图标与文本位置
- 子科技槽位 `sub_technology_slot_*` 位置

## 15.2 改框体底图样式

框体背景由 `quadTextureSprite` 或 `spriteType` 指向的 GFX 决定。  
常见状态底图：

- `GFX_technology_unavailable_item_bg`
- `GFX_technology_available_item_bg`
- `GFX_technology_researched_item_bg`
- `GFX_technology_currently_researching_item_bg`

在 `countrytechtreeview.gfx` 可替换对应 texture，完成统一换肤。

## 15.3 改整页背景样式

在页面容器内修改背景 icon 的 `spriteType`，如：

- `GFX_infantry_techtree_bg`
- `GFX_support_techtree_bg`

也可新增自定义 `GFX_pw_xxx_techtree_bg`。

## 15.4 改路径密度与观感

通过 `gridboxtype.slotsize` 控制网格密度：

- 更大 `slotsize`：节点间距更稀疏
- 更小 `slotsize`：节点更密集

通过 `gridboxtype.format` 控制绘制方向（如 `LEFT` / `UP`）。

## 16. Defines / Modifiers / Scopes 在科技树中的使用边界

## 16.1 Defines（全局开关，谨慎）

`common/defines/*.lua` 适合做全局规则调整，不建议用于单分支局部平衡。  
原则：先用技术效果与修饰符解决问题，最后才改 defines。

## 16.2 Modifiers（科技效果主体）

科技可通过多种修饰目标生效，例如：

- 国家层面修饰符
- 单位分类修饰符（如 `category_all_infantry`）
- 兵种层修饰符（按技术脚本支持范围）

建议将“属性变化”集中在技术块内，避免同一效果分散到事件/国策造成追踪困难。

## 16.3 Scopes（触发与效果作用域）

在 `available`、`allow`、`ai_will_do` 等逻辑中，需保证作用域正确。  
原则：

- 条件判断使用合法作用域对象
- 不在错误作用域调用效果/触发器
- 新增逻辑后必须通过日志验证

---

## 17. GUI 深度排错清单（实战）

出现“图标正常但科技不显示/错位/名字异常”时，按顺序排查：

1. **技术存在性**：`common/technologies` 是否定义该 ID
2. **根分支绑定**：是否存在 `<root_tech_id>_tree`
3. **folder 对齐**：技术 `folder.name` 与 GUI 容器名是否一致
4. **grid 坐标**：`gridboxtype.position` 是否跑出背景范围
5. **单位格尺寸**：`slotsize` 是否与技术坐标规模匹配
6. **项目模板**：该 folder 是否有 `techtree_<folder>_item` 模板
7. **图标映射**：是否有 `GFX_<tech_id>_medium`
8. **本地化映射**：是否有 `<tech_id>` 与 `<tech_id>_desc`
9. **开局科技**：`history/countries` 是否引用了已删除旧 ID
10. **日志关键字**：
   - `Found no grid box for tech`
   - `Unknown effect-type`
   - 本地化解析错误

## 18. 面向 Plentiful World 的落地建议

- 步兵页继续保持“全 PW 键值 + 全 PW grid 名”。
- 新增科技类型时，优先走“新 folder + 新 tab + 新模板”的完整流程，不在旧页面硬塞。
- 每次只改一个页面并进行日志验证，避免跨页面联动引入不可见回归。
- 对框体样式改造，先复制现有 `techtree_<folder>_item` 模板，再逐项调整，不直接在多页面共享模板上激进改动。

## 19. 情欲科技实现复盘（2026-03）

本节记录情欲科技页面与大图标适配的实战经验，后续同类页面直接复用。

## 19.1 科技卡模板的“硬绑定命名”

科技卡容器内以下节点名属于引擎硬绑定，不能随意删改或改名：

- `Background`
- `Icon`
- `bonus_icon`
- `bonus`
- `can_assign_design_team_icon`

若缺失，日志会出现 `Could not find ... in window techtree_xxx_item`，并伴随显示异常。

## 19.2 点击区域由状态背景 GFX 决定

科技卡可点击热区不是只看 `containerWindowType.size`，还受该 folder 的状态背景 sprite 影响。  
因此新 folder 必须补齐以下命名模式（normal/small）：

- `GFX_technology_<folder>_unavailable_item_bg`
- `GFX_technology_<folder>_available_item_bg`
- `GFX_technology_<folder>_researched_item_bg`
- `GFX_technology_<folder>_branch_item_bg`
- `GFX_technology_<folder>_currently_researching_item_bg`
- `GFX_technology_<folder>_small_*` 同套状态

未补齐时常见表现：视觉看起来是大框，但只在左上角小块可点击。

## 19.3 大图标显示策略（315x420）

- 科技页图标统一用 `GFX_<tech_id>_medium` 对应 315x420 资源。
- 图像预处理可选两种：
  - 拉伸填充（严格铺满，不留黑边）
  - 等比裁剪填充（保比例但裁切边缘）
- 本项目当前采用拉伸填充，以满足“无黑边优先”。

## 19.4 详情窗口（technology_info_window）关键点

- `tech_icon` 必须是详情窗口的直系节点；放进自定义子容器后，常见无法注入科技图。
- 装备科技要显示装备小图标，详情窗口需包含 `equipment_icon`（必要时 `carrier_fighter_icon`）节点。
- 研究按钮是否可见与层级顺序相关：若可滚动 `statsarea` 在后且覆盖范围过大，会遮挡按钮。
- 解决原则：
  - 将按钮区域放在 `statsarea` 之外且可见坐标
  - 控制 `statsarea` 起始 Y，避免覆盖按钮
  - 图标容器与文本容器做分区布局，避免相互遮挡

## 19.5 UI 元素叠放顺序规则

同一容器中，后定义元素会覆盖先定义元素。  
做大图与自定义框时建议顺序：

1. `Background`（状态底层）
2. 自定义框体/装饰层
3. `Icon`
4. 文本与角标
5. 交互附加标记（如设计团队图标）
